// SQL INJECTION LAB MEDIUM
SQL Injection (SQLi) happens when user-supplied data is concatenated directly into a SQL query. The attacker breaks out of the string context and injects arbitrary SQL, bypassing authentication or dumping the database. This simulated login form passes your input straight into a query string.
QUICK PAYLOADS: Auth bypass Comment bypass OR 1=1 Valid login Failed login

✓ SECURE FIX — Parameterised Queries

Never concatenate user input into SQL strings. Use prepared statements:
SELECT * FROM users WHERE username=? AND password=?
In Python (psycopg2): cursor.execute(query, (username, password))
Use an ORM (SQLAlchemy, Django ORM, Prisma) which parameterises automatically.
Apply least-privilege database accounts — the web app user shouldn't have DROP rights.

// CSRF ATTACK DEMO MEDIUM
Cross-Site Request Forgery (CSRF) tricks an authenticated user into unknowingly submitting a request. The browser automatically includes session cookies with every request to a domain — so a malicious site can silently submit forms to your bank, email provider, or any app you're logged into. This demo shows two simulated sites: a vulnerable bank and a malicious attacker page.

<form action="bank.example.com/transfer" method="POST">
 <input type="hidden" name="to" value="attacker">
 <input type="hidden" name="amount" value="9999">
 <!-- CSRF token required -->
</form>
<script>document.forms[0].submit()</script>

✓ SECURE FIX — Synchroniser Token Pattern

1. Generate a random, unpredictable CSRF token per session and embed it in every state-changing form.
2. On the server, verify the token matches before processing the request.
3. Also set SameSite=Strict or SameSite=Lax on session cookies — modern browsers will refuse to send them cross-site.
4. Use the Origin and Referer headers as a secondary check.
